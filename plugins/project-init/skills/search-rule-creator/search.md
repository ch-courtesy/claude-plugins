# 검색 지침

정보를 찾을 때 따라야 할 규칙입니다. 개별 명령·도구를 나열하는 대신, 검색 대상의 **종류와 범위**를 먼저 판단하고 거기에 맞는 방법을 선택합니다.

## 검색 대상 분류

작업 전에 아래 중 어디에 해당하는지 판단하고, 해당 카테고리의 규칙을 따릅니다.

### 1. 파일·경로 검색 (Name / path)

**특성**: 파일명·디렉토리 패턴으로 위치를 찾음.

**규칙**:
- 파일시스템 탐색 전용 도구(프로젝트에서 제공하는 Glob/파일 리스트 도구, `fd`, `find`)를 사용합니다.
- `ls -R | grep` 같은 파이프보다 전용 도구가 빠르고 의도가 분명합니다.
- 글로브 패턴을 명시적으로 작성합니다. 예: `src/**/*.ts`는 디렉토리 전체가 아닌 TS 파일만 매치.

### 2. 코드 내용 검색 (Content)

**특성**: 파일 안의 문자열·패턴·키워드를 찾음.

**규칙**:
- 콘텐츠 검색 전용 도구(`rg`/ripgrep, 프로젝트의 Grep 도구)를 우선 사용합니다. `grep -r`의 상위 호환.
- 식별자를 찾을 때는 단어 경계(`-w`, `\b`)로 부분 매치 노이즈를 줄입니다.
- `node_modules`, `dist`, `build` 같은 도구 관리 영역은 기본적으로 제외합니다. (파일시스템 지침의 "도구 관리 영역" 참조)
- 정규식은 필요할 때만 사용합니다. 리터럴 검색이 더 빠르고 오해가 적습니다.

### 3. 심볼·정의 검색 (Definition)

**특성**: 함수·클래스·변수가 정의된 위치를 찾음.

**규칙**:
- 언어 지원 도구(LSP, "Go to Definition", 언어별 인덱서)가 있으면 우선 사용합니다.
- 없으면 선언 패턴 중심의 grep으로 좁힙니다 (예: `^(def|function|class)\s+이름`).
- 정의와 호출처를 혼동하지 않도록 매치의 문맥을 확인합니다.

### 4. 로그·데이터 검색 (Logs / data)

**특성**: 로그·덤프·데이터 파일에서 특정 이벤트·값을 찾음.

**규칙**:
- 구조화된 로그(JSON 등)는 전용 쿼리 도구(`jq`, 로그 검색 시스템)를 사용합니다.
- 시간 범위로 먼저 좁히고, 그 안에서 내용 필터를 적용합니다. 전체 파일에 바로 내용 검색을 거는 것은 마지막 수단.
- 대용량 파일은 스트리밍으로 처리합니다. 전체 로드 금지.

### 5. 외부·웹 검색 (External)

**특성**: 프로젝트 밖의 문서·레퍼런스·예제를 찾음.

**규칙**:
- **1차 레퍼런스**(공식 문서, 표준 스펙, 원본 소스 코드)를 우선합니다. 블로그·커뮤니티 글은 보조로만.
- 참고한 자료의 **버전**과 현재 프로젝트의 버전이 다른지 먼저 확인합니다.
- 외부 정보를 적용하기 전 최소 한 번은 프로젝트 컨텍스트에서 검증합니다 (API 시그니처가 실제로 존재하는지, 의존성 버전이 맞는지 등).

## 전략 원칙

- **좁게 시작, 필요하면 넓히기**: 구체적인 쿼리로 먼저 시도하고, 비면 범위를 넓힙니다. 반대로 하면 노이즈가 폭발합니다.
- **전용 도구 > 셸 파이프라인**: Grep/Glob 같은 전용 도구가 있다면 우선 사용합니다. `cat | grep | awk` 체인은 피합니다.
- **이미 아는 것 재검색 금지**: 이미 읽은 파일에 답이 있으면 다시 검색하지 말고 참조합니다.
- **결과 재사용**: 같은 쿼리를 반복하게 되면 결과를 기록해 두고 재사용합니다.
- **빈 결과도 정보**: 검색이 비었다면 그 자체가 단서입니다. 쿼리를 바꿀지, 가정을 바꿀지, 대상을 바꿀지 판단합니다.
- **대규모 탐색은 위임**: 같은 주제로 3회 이상의 검색·읽기가 예상되면 탐색 전용 에이전트(Explore 등)나 병렬 작업으로 위임하는 것을 고려합니다.

## 공통 규칙

- 검색 전에 **무엇을, 왜** 찾는지 한 문장으로 정리합니다. 그것만으로 쿼리가 명확해지는 경우가 많습니다.
- 결과가 예상과 다르면 쿼리를 탓하기 전에 **가정**을 검토합니다 (이름이 바뀌었는지, 파일이 이동했는지, 이 프로젝트가 맞는지).
- 검색 결과는 **증거**입니다. 해석은 주변 맥락을 확인한 뒤에 합니다 — 한 줄 매치만 보고 결론을 내리지 않습니다.
