# 샌드박스 지침

위험하거나 영향 범위가 큰 작업을 격리된 환경에서 수행하는 규칙입니다. 여기서 "샌드박스"는 두 가지 맥락을 모두 포함합니다:

- **격리된 실행 환경** — 별도 브랜치·워크트리, 가상환경, 컨테이너, VM, 임시 디렉토리
- **도구의 실행 샌드박스** — Claude Code 등 에이전트 도구가 명령 실행 시 자동으로 적용하는 제한 모드

## 사용 판단 원칙

작업이 아래 특성 중 하나라도 해당하면 먼저 샌드박스에서 수행합니다. 애매하면 사용자에게 확인합니다.

- **비가역 영향**: 복구가 어렵거나 시간이 걸림 (대량 파일 수정, DB 마이그레이션, 히스토리 재작성).
- **외부 신뢰도 미확인**: 외부 스크립트나 출처가 불분명한 코드 실행.
- **파괴적 명령 대량 실행**: 일괄 `rm`, 일괄 정리 스크립트 등.
- **실험적 변경**: 결과를 본 뒤 반영 여부를 결정해야 하는 경우.

## 격리 수단 선택

작업 성격에 맞는 최소 충분 수준의 격리를 사용합니다.

### 1. 브랜치·워크트리 수준

**특성**: 코드 변경만 격리하면 충분함.

**규칙**: 새 브랜치 또는 `git worktree`로 분리해 작업. 실패 시 브랜치 폐기.

### 2. 가상환경 수준

**특성**: 의존성 설치·제거처럼 런타임을 바꾸는 작업.

**규칙**: 언어별 가상환경(`venv`, 프로젝트 로컬 Node/Python 등)을 사용. 전역 환경은 건드리지 않습니다.

### 3. 컨테이너·VM 수준

**특성**: 시스템 수준 변경, 신뢰할 수 없는 코드 실행, 파일시스템·네트워크 효과를 완전히 분리해야 하는 경우.

**규칙**: Docker 등 컨테이너 또는 VM을 사용합니다. 호스트 마운트는 필요한 최소 경로·읽기 전용을 기본으로.

### 4. 도구 제공 샌드박스

**특성**: Claude Code 등이 명령 실행에 자동으로 적용하는 파일시스템·네트워크 제한.

**규칙**:
- **기본적으로 샌드박스를 유지**합니다.
- 샌드박스 해제(예: `dangerouslyDisableSandbox`, 권한 확장)는 **사용자의 명시적 승인**이 있을 때만.
- 해제는 **필요한 단일 명령**에만 적용하고, 세션 전체에 유지하지 않습니다.

## 샌드박스 내부 규칙

- **상태 누출 차단**: 샌드박스에서 생성한 자격증명·토큰·캐시가 호스트로 새어나가지 않는지 확인합니다. 마운트 경로와 네트워크 정책을 점검.
- **임시 데이터 정리**: 실험 종료 시 컨테이너·워크트리·임시 디렉토리를 정리합니다. 정리 전 필요한 산출물은 명시적으로 호스트로 복사.
- **결과 반영은 명시적으로**: 샌드박스에서 나온 변경을 실제 프로젝트·프로덕션에 반영할 때는 diff를 검토하고, 커밋·머지·배포 절차를 분리된 단계로 수행합니다.
- **샌드박스 내부에서도 민감정보·파괴적 명령은 동일 주의**를 적용합니다. 격리는 영향을 줄일 뿐, 주의를 면제해 주지 않습니다.

## 해제 승인 기준

도구 샌드박스를 해제하기 전 다음을 모두 확인하고 사용자에게 제시합니다:

1. **필요성** — 해제 없이 달성할 방법이 정말 없는가.
2. **범위** — 한 번의 명령에만 한정되는가, 아니면 지속되는가.
3. **위험** — 어떤 외부 영향(네트워크, 시스템 파일, 다른 프로세스)이 발생할 수 있는가.
4. **검증** — 실행 후 결과를 어떻게 확인할 것인가.

네 항목 모두 명확하더라도 **사용자의 명시적 승인 없이는 해제하지 않습니다.**

## 공통 규칙

- 격리 수준 선택이 애매하면 **더 강한 쪽**으로 기울입니다(브랜치 vs 워크트리 → 워크트리, 가상환경 vs 컨테이너 → 컨테이너).
- 샌드박스 결과를 외부로 반영하기 전에 diff·로그·사이드 이펙트를 한 번 더 검토합니다.
- 실패한 샌드박스 환경을 방치하지 않습니다. 원인 파악 후 정리 또는 격리 상태 보존 중 하나를 선택합니다.
