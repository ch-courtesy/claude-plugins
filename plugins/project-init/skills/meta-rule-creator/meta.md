# 지침 유지보수 지침

`rules/` 하위 지침들을 건강하게 유지하기 위한 메타 규칙입니다. 지침은 한 번 작성하고 끝이 아니라 프로젝트와 함께 진화해야 합니다.

## 갱신 트리거

다음 상황에서 기존 지침을 업데이트하거나 새 규칙을 추가합니다:

- **반복되는 피드백**: 사용자가 같은 교정을 두 번 이상 제공했다면 규칙으로 승격합니다.
- **모호함·모순 발견**: 작업 중 기존 지침이 답을 주지 못했거나 서로 충돌했을 때.
- **새 도메인·도구 도입**: 기존 카테고리로 커버되지 않는 영역이 생겼을 때.
- **사건 이후**: 실수·장애가 발생한 뒤 같은 일을 반복하지 않기 위한 지침을 추가합니다.
- **성공한 비자명 선택**: 특이하지만 사용자가 명시적으로 승인한 접근도 규칙으로 남깁니다. 교정만 기록하면 지침이 점점 보수적·소극적으로 드리프트합니다.

## 캡처 대상

지침으로 남길 **가치가 있는 것**:

- **비자명한 결정**: 코드 읽기만으로는 파악되지 않는 "왜".
- **반복될 가능성**: 앞으로 다시 등장할 판단 기준.
- **값비싼 교훈**: 이미 한 번 대가를 치른 실수의 원인.
- **암묵적 합의**: 팀·프로젝트가 말하지 않고 공유하는 전제.

지침으로 **남기지 않을 것**:

- 코드에서 직접 파악 가능한 정보 (구조, API 시그니처, 파일 경로).
- 일회성 상황에만 적용되는 판단.
- 너무 구체적이어서 다른 맥락에서는 오히려 방해가 되는 규칙.

## 배치 기준

- 적합한 **카테고리 파일**(`rules/filesystem.md`, `git.md`, `bash.md`, `sandbox.md` 등)에 배치합니다.
- 기존 카테고리 어디에도 맞지 않고 **관련 규칙이 2개 이상** 쌓이면 새 카테고리를 만듭니다. 하나짜리 규칙 때문에 새 파일을 열지 않습니다.
- 여러 카테고리에 걸치는 규칙은 가장 핵심적인 카테고리에 두고 다른 카테고리에서는 링크로 참조합니다. 같은 내용을 중복 보관하지 않습니다.

## 작성 원칙

- **원칙 > 열거**: "이 경로들을 피하라"보다 "이런 특성의 파일은 이렇게 다룬다"로 씁니다. 목록은 예시로만 쓰고, 판단 기준을 본문에 담습니다.
- **Why 포함**: 각 규칙이 왜 존재하는지 짧게라도 남깁니다. 예외 상황에서 판단할 수 있도록.
- **간결함**: 필요한 것만 기록합니다. 매뉴얼이 아니라 결정 도구입니다.
- **검색 가능한 제목**: 섹션 제목은 나중에 찾기 쉬운 키워드로 정합니다.

## 검토와 정리

- **정기 검토**: 규칙이 여전히 맞는지 주기적으로 훑고, 맞지 않으면 수정·삭제합니다.
- **중복 통합**: 같은 내용이 여러 파일에 흩어져 있으면 한 곳으로 합칩니다.
- **상충 해결**: 규칙 간 충돌을 발견하면 즉시 둘 중 하나를 업데이트하거나 둘을 통합합니다.
- **죽은 규칙 제거**: 더 이상 해당 상황이 발생하지 않는 규칙은 남겨두지 않습니다. 파일만 무거워지고 신호가 약해집니다.

## 변경 절차

- 규칙 변경은 **코드 변경처럼 커밋**에 포함하고, 커밋 메시지에 "왜 이 규칙이 생겼는지·바뀌었는지"를 남깁니다.
- 큰 구조 변경(새 카테고리 생성, 카테고리 이름 변경, 대규모 재배치)은 사용자와 확인 후 진행합니다.
- 규칙 변경 후 기존 규칙을 참조하던 다른 파일·링크가 깨지지 않았는지 확인합니다.

## 공통 규칙

- 지침은 **프로젝트와 함께 진화**한다는 전제로 관리합니다. 완벽한 초기 구조를 만들기보다 필요할 때 개선하는 흐름을 유지합니다.
- 새 규칙을 추가할 때는 **이미 있는지 먼저 검색**합니다. 중복은 혼선을 만듭니다.
- 애매하면 사용자에게 "이걸 규칙으로 남길까요?"를 먼저 물어봅니다.
