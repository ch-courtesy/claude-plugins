# Git 지침

Git 작업 시 따라야 할 규칙입니다. 개별 명령을 나열하는 대신, 작업의 **영향 범위와 가역성**을 기준으로 분류하고 카테고리별 규칙을 적용합니다.

## 작업 분류

명령을 실행하기 전에 작업이 어디에 속하는지 판단하고, 해당 카테고리의 규칙을 따릅니다. 애매하면 사용자에게 확인합니다.

### 1. 로컬·가역 (Local, reversible)

**특성**: 로컬 레포에만 영향을 주고 쉽게 되돌릴 수 있음.

**규칙**: 필요에 따라 자유롭게 수행합니다. 스테이징, 새 커밋, 스태시, 로컬 브랜치 생성·전환.

**예**: `git add`, `git commit`(새 커밋), `git stash`, `git branch <name>`, `git checkout -b <name>`.

### 2. 공유·가역 (Shared, reversible)

**특성**: 원격이나 공유 상태에 영향을 주지만 일반적으로 되돌릴 수 있음.

**규칙**:
- 실행 전에 대상 브랜치와 영향 범위를 확인합니다.
- 공유 브랜치(`main`, `master`, `release/*`, 팀 개발 기본 브랜치)에 **직접 푸시하지 않습니다.** 변경은 PR을 통해 머지합니다.
- 푸시 직전 `git status`, `git log`로 의도한 커밋만 들어가는지 확인합니다.

**예**: 피처 브랜치 푸시, 원격 브랜치 생성, PR 생성·업데이트.

### 3. 파괴적 (Destructive)

**특성**: 이력이나 타인의 작업을 덮어쓸 수 있고 복구가 어렵거나 불가능함.

**규칙**:
- **사용자의 명시적 승인 없이 수행하지 않습니다.**
- 실행 전에 (a) 왜 필요한지, (b) 영향 범위, (c) 더 안전한 대안을 먼저 제시합니다.
- 공유 브랜치에 대한 파괴적 작업은 기본적으로 거부하고, 요청이 있어도 한 번 더 확인합니다.

**예**: `git push --force` / `--force-with-lease`, `git reset --hard`, `git clean -fd`, `git branch -D`, 공개된 커밋에 대한 `git rebase` / `cherry-pick --redo`, `git filter-*`, 히스토리 재작성, 태그 덮어쓰기.

## 커밋

- **새 커밋을 만듭니다.** 기본적으로 `--amend` 하지 않습니다. 사용자가 명시적으로 요청할 때만 amend.
- **스테이징은 명시적으로**: 파일 이름을 지정해 스테이징합니다. `git add -A` / `git add .`는 민감정보·임시 파일을 실수로 포함할 수 있어 주의합니다.
- **민감정보 파일이 스테이징에 포함되면** 커밋을 중단하고 사용자에게 알립니다. (파일시스템 지침의 "비밀·민감 정보" 카테고리 참조)
- **훅을 건너뛰지 않습니다.** `--no-verify`, 서명 비활성화(`--no-gpg-sign`) 등은 사용자의 명시적 요청이 있을 때만 사용합니다. 훅 실패는 우회하지 않고 원인을 수정합니다.
- **메시지 스타일**은 프로젝트의 기존 `git log` 스타일을 따릅니다. 확인되지 않으면 짧은 제목 한 줄 + 필요 시 빈 줄 후 본문(왜/무엇).

## 브랜치

- 기능·버그·작업 단위별로 브랜치를 분리합니다. 공유 브랜치에서 직접 작업하지 않습니다.
- **이름 규칙**은 프로젝트 컨벤션을 따릅니다. 확인되지 않으면 `<카테고리>/<짧은-설명>` 형태 사용 (예: `fix/login-timeout`, `feat/search`).
- 머지 완료된 로컬 브랜치 삭제는 사용자 확인 후에만 수행합니다.

## 원격·PR

- `git push` 시 대상 브랜치를 명시합니다. 상류(upstream)가 없으면 `-u`로 설정합니다.
- **`main` / `master`로 직접 푸시하지 않습니다.** 강제 푸시는 더더욱.
- PR 본문에는 변경 요약과 테스트 방법(또는 검증 절차)을 포함합니다.

## 훅 실패 대응

- 훅이 실패하면 **커밋이 성사되지 않은 상태**입니다. 이 상태에서 `--amend`는 이전 커밋을 수정하게 되므로 하지 않습니다.
- 원인을 수정 → 다시 스테이징 → **새 커밋**을 만듭니다.

## 공통 규칙

- 작업 전에 `git status`와 현재 브랜치를 확인합니다.
- 예상과 다른 상태(낯선 커밋, 예상 외 브랜치, 충돌)를 발견하면 덮어쓰거나 정리하지 말고 먼저 원인을 조사한 뒤 사용자에게 보고합니다.
- `git config`를 사용자 지시 없이 변경하지 않습니다.
