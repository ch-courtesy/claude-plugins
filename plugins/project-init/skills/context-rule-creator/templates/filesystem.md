# 컨텍스트 관리 지침

AI 에이전트 간에 작업 컨텍스트를 전달하기 위한 영속성 규칙입니다. 휘발 메모리(세션·대화)에 의존하지 않고, 한 에이전트가 끝낸 작업을 다른 에이전트가 이어받을 수 있어야 합니다.

이 프로젝트는 **파일시스템 백엔드**를 사용합니다. 모든 태스크는 `.tasks/` 아래에 마크다운 파일로 저장하고 git에 커밋합니다.

## 태스크 우선 원칙 (no-task-no-work)

비자명한 작업은 **관련 태스크 파일이 존재한 상태에서만 시작합니다**. 없으면 작업을 멈추고 먼저 태스크를 만듭니다 — `backlog`는 제목 + 1-2줄 설명만으로 캡처 가능하므로 비용은 매우 낮습니다. "작은 작업이라"·"즉흥적이라"는 면제 사유가 아닙니다.

### 작업 시작 신호 (이 중 하나라도 발생하면 태스크 존재 확인 필수)

- 첫 `Edit`/`Write` 호출
- `git add`/`git commit` 호출
- 외부 상태를 바꾸는 도구 호출 (외부 API, 인프라 변경 등)
- 한 카테고리에서 다파일 변경을 시작할 때

신호 발생 시 태스크가 없으면 즉시 작업 중단 → `.tasks/T-NNN.md` 생성 → 작업 재개. 사용자에게 "태스크가 없어 먼저 만들었습니다"라고 알립니다.

### 예외

- 사용자가 명시적으로 "이번엔 태스크 없이"·"룰 무시"처럼 면제해준 경우.
- 단순 정보 조회(파일 read, `git log`, 검색)는 작업 시작 신호가 아닙니다.

### 계획 섹션 채움

- **`backlog`는 계획이 비어 있어도 됩니다.** 제목 + 1-2줄로 아이디어를 캡처해 즉시 영구 저장소에 남기기 위함입니다.
- **`in_design` 또는 `in_progress`로 전이할 때는 계획 섹션이 채워져 있어야 합니다.** 본문의 계획 섹션(목표·배경·제안·검증 계획·완료 기준)을 다른 에이전트가 이해할 수 있을 만큼 채웁니다. 빈 섹션이나 `TBD`로 전이하지 않습니다.
- 사소한 태스크라도 각 섹션은 짧게라도 명확히 작성합니다. 분량이 짧은 것은 괜찮지만 비어 있는 것은 안 됩니다.
- 시작한 뒤 알게 된 사항은 진행 로그·결정 섹션에 추가하고, 필요하면 계획 섹션도 갱신합니다.

### 누락 발견 시

작업이 이미 진행되었는데 관련 태스크가 없음을 인지·지적당하면, 즉시 작업을 멈추고 retroactive 태스크 파일을 만듭니다. 핸드오프 메모와 진행 로그에 retroactive 사유를 기록한 뒤 작업을 재개합니다. 같은 세션 내에서 동일 누락을 두 번째 반복하지 않도록 합니다.

## 태스크 파일

- 위치: `.tasks/T-NNN.md`
- ID: `T-NNN` 일련번호. 한 번 부여하면 바꾸지 않습니다. 빈 번호는 재사용하지 않습니다.
- 한 파일 = 한 태스크. 디렉토리 중첩 없이 평평하게 둡니다.
- 부모-자식 관계는 frontmatter `parent:`로만 표현합니다.

### Frontmatter 스펙

```yaml
---
id: T-001
title: <짧은 제목>
status: in_progress       # backlog | in_design | in_progress | blocked | review | done | cancelled
parent: T-000             # 상위 태스크 ID (없으면 생략)
depends_on: [T-002]       # 차단 의존 (없으면 생략)
owner: <agent-or-user>    # in_progress·blocked일 때만 채움 (선택)
created: 2026-04-28T10:00:00Z
refs:                     # 태스크 전체와 연관된 핵심 산출물 1~3개 (없으면 생략)
  - path/to/file.ts
  - https://github.com/.../pull/42
---
```

수정 시각·마지막 활동은 git 메타데이터(`git log`)로 확인합니다. frontmatter에 별도 필드를 두지 않습니다.

### 본문 섹션 (순서 고정)

```markdown
## 핸드오프 메모
가장 최근 인계 사항. 다음 에이전트가 이어받을 때 알아야 할 것 (현재 상태·다음 단계·주의점).
태스크 생성 시점에는 비어 있어도 되고, 인계가 일어날 때마다 이 섹션을 갱신합니다.

## 목표
무엇을 달성하려는지. 결과 상태를 측정·확인 가능한 형태로 1~3 문장.

## 배경
왜 필요한지. 어떤 문제·상황에서 출발했고 무엇이 걸려 있는지. 처음 보는 사람도 맥락을 잡을 수 있을 만큼.

## 제안
어떤 접근·단계·대안으로 목표를 달성할지. 구현 순서, 사용할 도구·라이브러리, 검토한 대안과 선택 이유. 다른 에이전트가 받아 들었을 때 자체 판단 없이 이어갈 수 있을 만큼.

## 검증 계획
결과를 어떻게 확인할지. 사용할 테스트·도구·관찰 지표·수동 단계. DoD의 각 항목을 어떤 방법으로 확인하는지 매핑.

## 완료 기준 (Definition of Done)
- [ ] ...
- [ ] ...

## 진행 로그 (append-only)
- 2026-04-28T10:00:00Z @agent-a: 시작. ...
- 2026-04-28T11:00:00Z @agent-a: ... 발견.

## 결정 (Decisions)
- 2026-04-28: X 대신 Y 선택. 이유: ...
```

- **핸드오프 메모를 가장 위**에 둡니다. 인계받는 에이전트가 가장 먼저 읽어야 하기 때문입니다.
- **계획 섹션(목표·배경·제안·검증 계획·DoD)은 태스크 생성 시 모두 채웁니다.** 빈 채로 작업을 시작하지 않습니다.
- **목표·제안·DoD는 다른 층위입니다**: 목표는 "무엇을(What)", 제안은 "어떻게(How)", DoD는 "끝났는가(Done?)"의 체크리스트.
- **검증 계획과 DoD는 분리합니다**: DoD는 "끝났는가"의 체크리스트, 검증 계획은 그 체크리스트를 "어떻게 확인하는가"의 방법론.
- **진행 로그에 적는 것**: 결정·차단 사유·놀라운 발견·외부 영향이 있는 변경. 모든 도구 호출이나 사소한 시도까지는 적지 않습니다 (잡음 폭증 방지).
- **참조(refs) 위치**: frontmatter `refs:`는 태스크 **전체와 연관된 핵심 산출물**(주요 PR, 핵심 파일 1~3개)만. 진행 중 등장하는 임시 참조나 부수 파일은 진행 로그 항목 안에서 언급합니다.

## 상태 어휘

`backlog` → `in_design` → `in_progress` → `review` → `done`

- `backlog`: 캡처만 된 상태 (계획 비어도 OK).
- `in_design`: 계획 작성 중. 본격 실행 전 단계.
- `in_progress`: 실행 중.
- `review`: 검토 중 (예: PR 리뷰).
- `done`: 완료.
- `blocked`: 차단 사유로 일시 정지. 해소되면 직전 상태로 복귀합니다.
- `cancelled`: 사유를 진행 로그에 기록합니다.

## 운영 규칙

1. **시작 전**: 관련 태스크 존재 확인. 없으면 작업을 멈추고 새 태스크를 먼저 만듭니다. 대상 파일을 읽고 **핸드오프 메모(본문 상단)를 가장 먼저** 확인합니다.
2. **시작 시**: `status: in_progress`로 바꾸고 `owner`를 본인으로 설정합니다. 진행 로그에 시작 항목을 추가합니다.
3. **진행 중**: 의미 있는 발견·결정·차단을 **즉시** 진행 로그·결정 섹션에 기록합니다. 묶어서 미루지 않습니다. 계획 섹션이 변경되면 그 섹션도 갱신합니다.
4. **commit 전**: 태스크 본문의 "검증 계획"에 명시된 항목 중 자동화 가능한 것은 commit 전에 모두 실행해 통과를 확인합니다. 통과 흔적은 commit message 또는 진행 로그 항목으로 남깁니다. 미통과 시 commit 중단 → 원인 수정 → 재검증. **검증 미실행·미통과 상태로 commit하지 않습니다 — "급하다"·"작은 변경이라"는 면제 사유가 아닙니다.** 사용자 환경이 필요한 수동 검증 항목은 핸드오프 메모에 누가·언제 진행할지 명시합니다.
5. **완료 시**: 검증 계획에 따라 DoD 점검 → `status: review` 또는 `done` → 핸드오프 메모 갱신 → `owner` 비움.
6. **append-only**: 진행 로그·결정 섹션은 덮어쓰거나 삭제하지 않습니다. 정정은 새 항목으로 추가합니다.

## 태스크 단위

- 1 태스크 ≈ 한 에이전트가 한 세션에 끝낼 수 있는 분량.
- 더 크면 하위 태스크로 분할합니다. 부모 태스크는 하위 태스크의 진행 상황을 요약해 둡니다.
- 깊이는 2~3 단계 이내를 권장합니다.

## 공통 규칙

- 태스크 파일은 git에 커밋합니다. 다른 에이전트와 사람이 PR·리뷰에서 컨텍스트를 따라잡을 수 있어야 합니다.
- 시각은 ISO 8601 UTC를 사용합니다 (예: `2026-04-28T10:00:00Z`).
- ID는 한 번 부여하면 바꾸지 않습니다. 잘못 만들었으면 `cancelled` 처리하고 새 ID로 다시 만듭니다.
