# 도메인 지침

프로젝트의 비즈니스·도메인 지식을 다루는 규칙입니다. 도메인 내용 자체(용어·규칙·엔티티)는 **프로젝트마다 다르므로**, 이 문서는 "어떻게 기록하고 어떻게 적용할지"에 대한 메타 규칙이고 실제 내용은 `rules/domain/` 하위의 주제별 파일로 분리해 보관합니다.

## 문서 구성

`rules/domain/` 하위에 주제별 파일을 두고 관리합니다. 해당 없는 주제는 생략하고, 프로젝트 특유의 주제가 있으면 추가합니다.

- **`glossary.md`** — 용어집. 정확한 의미, **의도하지 않는 의미**, 코드에서의 표기.
- **`entities.md`** — 주요 개체와 속성, 관계, 식별자 규칙.
- **`invariants.md`** — 언제나 성립해야 하는 제약 (예: 잔고 ≥ 0, 주문 상태는 단방향).
- **`lifecycle.md`** — 상태와 전이, 허용된 전이와 금지된 전이.
- **`policies.md`** — 비즈니스 규칙. 조건과 결과를 명확히.
- **`stakeholders.md`** — 사용자·역할·권한, 각 역할이 할 수 있는 일.
- **`integrations.md`** — 외부 시스템과의 계약 (입력·출력, 에러 의미, 타임아웃 기준).
- **`edge-cases.md`** — 알려진 특이 케이스와 그 이유.

## 캡처 원칙

도메인 지식을 기록할 때 따릅니다.

- **용어 일치**: 코드 식별자와 도메인 용어를 동일하게 유지합니다. 별명·번역 층은 동기화 비용을 만듭니다. 불가피하면 이유를 glossary에 명시합니다.
- **의도하지 않는 의미 기록**: "X는 A가 아니다"가 "X는 B다"만큼 중요합니다. 용어 오해를 막기 위해 부정적 정의도 남깁니다.
- **불변식은 타입·테스트로도**: 말로만 적지 말고 가능한 한 타입 시스템·검증 로직·테스트로 실행 가능한 형태로 표현합니다. 문서는 보조, 코드가 1차 진실입니다.
- **결정권자 명시**: 각 정책·규칙 옆에 누가·언제 결정했는지를 기록합니다. 나중에 변경·재확인할 때 필요합니다.

## 적용 원칙

작업 중 도메인 관련 결정이 필요할 때 따릅니다.

- **추측 금지**: 비즈니스 규칙이 문서에 없거나 모호하면 **사용자에게 확인**합니다. 개발자의 기술적 판단으로 정하지 않습니다.
- **기술 결정 ≠ 도메인 결정**: "성능상 유리"는 기술 결정이지만, "결제 실패 시 자동 환불"은 도메인 결정입니다. 둘을 혼동해 결정권을 넘지 않습니다.
- **우선순위**: 비즈니스 규칙이 기술 편의보다 우선합니다. 규칙이 구현하기 어렵다면 구현 방법을 바꾸지 규칙을 단순화하지 않습니다.
- **엣지 케이스는 질문으로**: "이 경우는 어떻게 처리해야 하나요?"를 적극적으로 묻습니다. 모르면 기본값을 정하지 말고 확인합니다.

## 진화 관리

- 도메인은 비즈니스와 함께 변합니다. 규칙이 바뀌면 문서도 함께 갱신합니다.
- **용어 변경**: glossary 갱신 + 코드 식별자 일괄 변경 + 마이그레이션 플랜을 함께 검토합니다.
- **불변식 변경**: 특히 신중하게. 이미 위배된 기록이 데이터에 남아 있을 수 있으므로 백필·마이그레이션을 먼저 계획합니다.
- 규칙·문서 변경의 근거(요청자, 시점, 이유)를 커밋 메시지에 남깁니다. (메타 지침 참조)

## 코드와의 관계

- **이름짓기**: 파일·클래스·함수·변수 이름은 도메인 용어를 반영합니다. `processItem()`보다 `refundOrder()`가 의미를 전달합니다.
- **모델 분리**: 외부 시스템의 모델(DB row, API payload)과 도메인 모델을 구분합니다. 둘을 직접 이어붙이면 외부 변경이 도메인 모델을 오염시킵니다.
- **테스트의 언어**: 테스트 이름과 assertion 메시지도 도메인 용어로 작성합니다. `test_user_flow_1`보다 `test_refund_within_30_days_issues_full_amount`.

## 공통 규칙

- 애매하면 **기록하고 물어봅니다.** 머릿속에만 있는 도메인 지식은 세션이 끝나면 사라집니다.
- 문서와 코드가 충돌하면 **코드를 기준**으로 재확인합니다. 문서를 갱신하거나 코드를 수정해 일관성을 회복합니다.
- 도메인 지식은 컨텍스트 계층 중 **프로젝트** 계층에 속합니다. 세션이나 영속 메모리가 아니라 `rules/domain/`에 보관합니다. (컨텍스트 지침 참조)
