# 보안 지침

코드·데이터·운영을 다룰 때 따라야 할 보안 규칙입니다. 개별 공격·취약점을 나열하는 대신, **데이터 민감도**와 **신뢰 경계**를 기준으로 분류하고 카테고리별 규칙을 적용합니다.

## 데이터 민감도 분류

정보를 다루기 전에 아래 중 어디에 해당하는지 판단합니다.

### 1. 비밀 (Secret)

**특성**: 자격증명·키·토큰·비밀번호. 노출되면 즉시 시스템 신뢰가 붕괴됨.

**규칙**:
- 코드·설정 파일·로그·URL·커밋 히스토리에 절대 포함하지 않습니다.
- 환경변수 또는 시크릿 매니저에서만 로드합니다. 하드코딩 금지.
- 메모리에 필요 이상으로 오래 유지하지 않습니다.
- 예시 파일(`.env.example` 등)에는 placeholder만 둡니다.

**예**: API 키, DB 비밀번호, JWT 서명 키, OAuth 시크릿, SSH/TLS 개인키.

### 2. 기밀 (Confidential)

**특성**: 개인정보(PII), 결제 정보, 영업 비밀. 노출되면 사용자·조직에 실질적 피해.

**규칙**:
- **최소 권한** 접근. 접근 로그를 남깁니다.
- 저장 시 암호화, 전송 시 TLS.
- 로그·에러 메시지에 출력할 때는 마스킹(예: `user@***.com`, 카드 뒷 4자리만).
- 디버그용이라도 평문 덤프 금지.

### 3. 내부 (Internal)

**특성**: 시스템 구성, 내부 메트릭, 소스 코드 등 외부 공개 대상은 아니지만 치명적이지는 않은 정보.

**규칙**: 신뢰된 네트워크·접근 제어 내에서만 노출합니다. 공개 저장소에 올리기 전에 검토.

### 4. 공개 (Public)

**특성**: 공개 문서, 오픈소스, 마케팅 자료.

**규칙**: 제한 없음. 단, 공개 내용이 **다른 계층의 비밀을 드러낼 단서**가 되는지 확인(예: 에러 메시지가 내부 구조 노출).

## 신뢰 경계

### 외부 입력 (Untrusted)

**특성**: 사용자 입력, 외부 API 응답, 파일 업로드, 쿼리 파라미터, 환경변수 값. 공격 면(attack surface).

**규칙**:
- **경계에서 검증**합니다. 타입·범위·길이·화이트리스트. 블랙리스트는 마지막 수단.
- **인젝션 방지**: SQL은 파라미터 바인딩, 셸 명령은 인자 배열 전달, HTML 렌더는 이스케이프, 파일 경로는 정규화·샌드박스.
- **역직렬화 주의**: 신뢰할 수 없는 입력을 `pickle`·`eval`·`Marshal` 계열로 복원하지 않습니다.
- **파일 업로드**: 확장자·MIME 검증, 실행 권한 제거, 저장 경로는 원본 이름과 분리.

### 외부 시스템 호출

**특성**: 우리 코드가 외부 서비스·API를 호출하는 방향.

**규칙**:
- **TLS 필수**: `http://`, verify 비활성화 금지.
- **타임아웃·재시도 제한**: 응답 없음으로 인한 자원 고갈 방지.
- **최소 권한 자격증명**: 전체 권한 토큰 대신 작업에 필요한 스코프만.
- **응답 검증**: 외부 응답도 "외부 입력"과 동일하게 신뢰하지 않습니다.

### 내부 호출 (Trusted)

**특성**: 우리 시스템 내부에서의 호출.

**규칙**: 계약을 신뢰합니다. 과도한 검증으로 코드를 복잡하게 만들지 않습니다. 단, **서비스 경계**(마이크로서비스 간 호출)는 외부 호출 수준으로 취급합니다.

## 인증·인가

- **인증(Authentication) ≠ 인가(Authorization)**: "누구인지"와 "무엇을 할 수 있는지"를 분리합니다.
- **서버 측 권한 체크**: 클라이언트·UI의 권한 체크는 UX용일 뿐, 실제 방어선은 서버입니다.
- **세션·토큰 수명 제한**: 장기 토큰은 탈취 시 피해가 큽니다. 리프레시·만료·리보크 체계를 둡니다.
- **민감 액션은 재인증**: 비밀번호 변경, 결제, 데이터 삭제는 세션 유효만으로 부족합니다.

## 감사·로깅

- **로그 금지 항목**: 비밀, PII, 토큰, 카드번호, 세션 ID 전체. 마스킹하거나 생략합니다.
- **보안 이벤트 기록**: 로그인 실패, 권한 거부, 이상 패턴, 비밀 접근은 명시적으로 남깁니다.
- **로그는 누구나 본다는 전제**: 로그 파일·스트림은 개발자·운영자·때로는 외부 감사자가 열람합니다. 그 가정으로 씁니다.

## 에러 응답

- **외부에는 일반화된 메시지**: `Internal server error`, `Unauthorized`. 스택 트레이스·SQL·경로·라이브러리 버전을 노출하지 않습니다.
- **상세는 서버 로그**: 개발자가 디버깅할 수 있는 맥락은 내부 로그에.
- **차이로 정보 누설 방지**: "존재하지 않는 사용자" vs "비밀번호 틀림"처럼 에러 메시지 차이가 열거 공격의 단서가 됩니다.

## 의존성·공급망

- **lockfile 커밋**: 재현 가능한 빌드. 업데이트는 명시적 액션으로만.
- **취약점 스캔**: 의존성 업데이트 시점 + 주기적으로 실행. 알려진 CVE가 있으면 평가 후 업데이트.
- **추가 전 검토**: 새 의존성은 유지보수 상태, 작성자 신뢰도, 필요성을 확인합니다.
- **`curl | sh` 금지**: 출처가 불분명한 원격 스크립트를 바로 실행하지 않습니다.

## 저장 보안

- **비밀 저장 우선순위**: 시크릿 매니저 > 환경변수 > 암호화된 설정 파일 > 평문. 코드 내 저장은 금지.
- **백업도 동일 수준**: 프로덕션 DB가 암호화되어 있으면 백업도 암호화되어야 합니다.
- **접근 로그 보존**: 누가 언제 어떤 자원에 접근했는지 추적 가능해야 합니다.

## 사건 대응

- **비밀 노출 시**:
  1. 즉시 해당 자격증명을 회전(rotate)합니다.
  2. 노출된 커밋을 제거하더라도 **이미 노출된 것으로 간주**합니다 (히스토리·미러·백업·로그에 남음).
  3. 유효했던 기간 동안 악용 흔적이 있는지 로그 점검.
- **취약점 발견 시**: 공개 채널 금지. 비공개 채널(보안 이메일, 담당자 DM)로 보고합니다.
- **데이터 유출 시**: 기술 대응 전에 **법적·컴플라이언스 통지 요구사항**(GDPR, 개인정보보호법 등)을 확인합니다.

## 공통 규칙

- **최소 권한 (Least privilege)**: 필요한 것만, 필요한 시간만.
- **심층 방어 (Defense in depth)**: 한 계층이 뚫려도 다음 계층이 막습니다. 단일 방어선에 의존하지 않습니다.
- **애매하면 차단**: 허용 여부가 불확실하면 기본은 차단. 허용은 명시적 결정을 거쳐 추가합니다.
- **"일어날 수 없는 일은 없다"**: 공격자는 예상 밖 경로로 움직입니다. `assert` 기반의 보안 가정은 실패합니다.
- **보안은 사후 추가 불가**: 기능 완성 후 덧붙이는 보안은 허점이 많습니다. 설계 시점부터 경계와 권한을 고려합니다.
- **불확실하면 보안 담당·사용자에게 확인**: 혼자 판단하지 않습니다.
