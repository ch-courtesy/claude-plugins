# 엔지니어링 지침

코드를 작성하고 변경할 때 따라야 할 규칙입니다. 개별 언어·프레임워크 관례를 나열하는 대신, 어떤 기술 스택에도 적용되는 **원칙**으로 정리합니다.

## 작업 범위

- **필요한 최소만 변경**: 버그 수정은 주변 리팩터링을 끌어오지 않습니다. 일회성 작업은 헬퍼·프레임워크를 만들지 않습니다.
- **스코프 크립 금지**: 관련 없는 개선은 별도 작업으로 제안하고, 현재 작업에 끼워 넣지 않습니다.
- **미래를 위한 설계 금지**: "나중에 필요할 것 같아서"는 근거가 아닙니다. 세 번째 반복이 보일 때 추상화합니다.
- **완성도**: "거의 동작"하는 구현을 남기지 않습니다. 반쯤 구현된 채로 커밋하지 않습니다.

## 추상화

- **세 번 규칙**: 같은 패턴이 세 번째 등장할 때 추상화를 고려합니다. 두 번은 우연, 한 번은 필연일 수 있습니다.
- **가짜 추상화 금지**: 한 번만 호출되는 helper, 하나의 구현체만 가지는 인터페이스는 오히려 가독성을 떨어뜨립니다.
- **중복은 잘못된 추상화보다 낫다**: 잘못 묶인 추상화는 이후 요구사항이 분기할 때 코드를 뒤틀어 놓습니다.
- **공개 API 확장은 신중히**: 한 번 공개하면 되돌리기 어렵습니다. 내부로 둘 수 있으면 내부에 둡니다.

## 에러 처리

- **시스템 경계에서 검증**: 사용자 입력, 외부 API, 파일 입력은 의심합니다. 내부 호출은 계약을 신뢰합니다.
- **일어날 수 없는 케이스 처리 금지**: 프레임워크·타입 시스템이 이미 보장하는 상태에 대한 방어 코드를 넣지 않습니다.
- **에러 삼킴 금지**: `try/except`로 예외를 묵살하지 않습니다. 처리할 수 있는 것만 잡고, 나머지는 전파합니다.
- **폴백의 비용**: 폴백은 문제를 숨깁니다. 근본 원인을 먼저 찾고, 폴백이 정말 필요한지 판단합니다.
- **에러 메시지는 수신자 기준**: 로그는 운영자가, 응답은 사용자가 읽습니다. 각 수신자에게 의미 있는 정보만 담습니다.

## 네이밍

- **도메인 언어 반영**: 식별자는 도메인 용어를 그대로 사용합니다. (도메인 지침 참조)
- **축약 지양**: 에디터의 자동완성을 신뢰합니다. `usr`보다 `user`, `cnt`보다 `count`.
- **의도 중심**: 이름이 **무엇을 하는지** 설명하게 합니다. `getUserById`가 `fetch`보다 낫습니다.
- **역할 일치**: 함수는 동사, 변수·타입은 명사. Boolean은 `is`/`has`/`can` 접두사로 명확히.

## 주석·문서화

- **기본값: 주석 없음.** 코드와 이름이 말해주는 것은 주석으로 반복하지 않습니다.
- **WHY만 남기기**: 주석은 코드가 드러내지 못하는 "왜"에 한정합니다 — 비자명한 제약, 숨겨진 불변식, 특정 버그 회피 이유.
- **WHAT / HOW 주석 금지**: 이름이 이미 말하는 동작을 다시 설명하지 않습니다.
- **현재 작업 참조 금지**: "#123을 위해 추가", "X 플로우에 사용됨" 같은 주석은 PR 설명·커밋 메시지에 속합니다. 코드베이스가 진화하면 낡아 거짓이 됩니다.
- **공개 API 문서**: 외부에서 호출되는 함수·모듈은 간결한 사용 예와 계약을 문서화합니다.

## 테스트

- **행동 기반**: 구현 상세가 아닌 외부에서 관찰 가능한 행동을 검증합니다. 내부 구조가 바뀌어도 테스트가 깨지지 않게.
- **도메인 용어로 기술**: 테스트 이름과 assertion이 도메인 언어를 쓰게 합니다. (도메인 지침 참조)
- **골든 패스 + 중요한 엣지**: 모든 경우를 테스트하지 않습니다. 가장 많이 쓰이는 경로와 실패 비용이 큰 엣지 케이스 위주.
- **브리틀 테스트 회피**: 시간·랜덤·순서에 의존하는 테스트는 관리 비용이 큽니다. 필요하면 격리·모킹으로 결정적으로 만듭니다.
- **테스트 자체도 코드**: 중복·마법 숫자·이해 어려운 설정은 테스트에서도 정리합니다.

## 리팩터링

- **버그 수정과 리팩터링 분리**: 두 가지를 한 커밋에 섞지 않습니다. 리뷰어가 의도를 구분할 수 있어야 합니다.
- **동작 불변 보장**: 리팩터링은 행동을 바꾸지 않습니다. 테스트가 모두 통과하는 상태에서 시작하고 끝냅니다.
- **큰 리팩터링은 단계별로**: 한 번에 쏟아붓지 않습니다. 중간 커밋마다 시스템이 동작해야 합니다.
- **우연히 발견한 지저분함**: 현재 작업과 무관하면 메모만 남기고 별도 작업으로 처리합니다.

## 작업 완료 확인

- **빌드 성공 ≠ 기능 동작**: 타입 검사·린트·단위 테스트 통과는 "코드가 돌아간다"의 증거일 뿐, "기능이 올바르다"의 증거가 아닙니다.
- **UI 변경**: 개발 서버를 띄우고 실제 브라우저에서 사용해 봅니다. 골든 패스와 엣지 케이스 모두. 테스트가 불가능하면 그렇게 명시합니다.
- **통합 지점**: 외부 서비스·DB와 연결되는 코드는 로컬에서 한 번은 실제 연결로 돌려 봅니다.
- **회귀 확인**: 이 변경이 다른 기능을 건드리지 않았는지 관련 주요 경로를 점검합니다.

## 공통 규칙

- **단순함이 기본**: 세 줄의 중복은 미숙한 추상화보다 낫습니다.
- **읽는 사람 중심**: 코드는 쓰기보다 읽히는 시간이 훨씬 깁니다. 작성자의 편의보다 독자의 이해를 우선합니다.
- **불확실하면 물어봄**: 요구사항·제약·스타일이 불명확하면 추측하지 않고 확인합니다.
